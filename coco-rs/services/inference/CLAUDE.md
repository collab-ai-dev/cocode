# coco-inference

Thin multi-provider LLM client wrapper over `vercel-ai`. Generic retry, usage aggregation, cache-break detection, thinking-level conversion — and **nothing Anthropic-specific** (auth/OAuth/prompt-cache/rate-limit policy live in `vercel-ai-anthropic`, not here).

**Not in scope here** (belong in `vercel-ai-anthropic` or are Anthropic-only concerns): OAuth, policy limits, rate-limit messaging, Claude.ai limits, auth helpers, beta resolution.

## Key Types

- `ModelRuntimeRegistry`, `ModelRuntime`, `QueryParams`, `QueryResult` — unified LLM call surface over `Arc<dyn LanguageModelV4>` slots
- `RetryConfig` — generic cross-provider retry; `UsageAccumulator` — token/cost accumulation
- `CacheBreakDetector`, `CacheBreakResult`, `CacheState` — prompt-cache boundary detection
- `StreamEvent`, `synthetic_stream_from_content` — streaming primitives
- `InferenceError`, `ErrorLog`, `RequestLog`, `ResponseLog`, `KnownGateway`
- `InferenceError::RateLimited.long_context_credits_required` — provider-neutral flag set from adapter-owned diagnostics when Anthropic rejects 1M context for missing credits; raw message matching stays in `vercel-ai-anthropic`
- DTO names (Message, content parts, `StopReason`, FinishReason, Usage, ProviderOptions, ProviderMetadata, ReasoningLevel) **are not re-exported here** — they live in `common/llm-types` (DTO seam). This crate owns runtime types only
- `merge_provider_options`, `provider_base_options`; `generate_tool_schemas`, `filter_schemas_by_model`, `estimate_schema_tokens`
- `cache_convert::to_extra_body` / `session_context_to_extra_body` — provider-neutral pass-through emission of `cacheStrategy` / `requestedBetas` / `agenticQuery` / `querySource`
- `build_call_options_with_extra` — returns `(LanguageModelV4CallOptions, BTreeMap<String, Value>)` so the cache-break detector hashes the merged map directly
- `prompt_layout::{SystemPromptPart, put_system_prompt_parts, build_prompt_layout_from_prompt}` — consumes the prompt-parts envelope on the System message (`provider_options["prompt_layout"].system_parts`). Anthropic gets one `system[]` block per part with `CacheHint::Breakpoint` → `cache_control: {type: "ephemeral"}`; OpenAI/Gemini flatten into native instruction slots. Cache-break hashing derives from that final layout
- `ProviderClientFingerprint` — includes `runtime_state_digest` over `account` + `prompt_cache` + per-provider `provider_options`; a settings-reload flip triggers a turn-boundary client rebuild, scoped per provider instance
- `QueryParams.fallback_min_context_window` — optional per-call floor; `ModelRuntime` skips fallback slots whose known `ModelInfo.context_window` is smaller (slots without `ModelInfo` stay eligible)

## Call path — bypasses `vercel-ai/ai` SDK layer

[`ModelRuntime::query_once`] / [`ModelRuntime::open_stream`] call `do_generate` / `do_stream` **directly** on the `Arc<dyn LanguageModelV4>` adapter. coco-rs does NOT route through `vercel_ai::generate_text` / `stream_text` in production (only `vercel-ai/ai/tests/live/` reaches them). Anything inside `vercel-ai/ai/src/generate_text/` is **dead for coco-rs**.

Tool-input handling lives in three layers, each owning a distinct concern:

- **Wire parsing — provider adapters + inference seam.** Adapters carry only wire strings and metadata; `services/inference` records parsed/raw state on `ToolCallSegment.input_state` (`ToolInputWireState`) so `app/query` never reparses. Empty/whitespace → `Empty`; clean or repaired JSON → `ParsedJson(Value)`; provider-marked unrecoverable raw → `UnrecoverableRaw { raw, error }` (copied to `ToolInputInvalidReason::JsonParseFailed`); possibly-legitimate freeform payloads → `RawStringAllowed { raw }`, left for tool-runtime coercion.
- **Schema validation — `app/query/src/tool_input_validate.rs`.** `validate_tool_call` runs `Value::String` recovery for calls not already wire-invalid, then JSON Schema validation via `coco_tool_runtime::ToolSchemaValidator` (pre-PreToolUse for raw input; `validate_effective_input_or_complete_error` in `tool_call_preparer.rs` catches hook-rewritten input). Sets `ToolCallPart.invalid_reason` to the structured variant (`SchemaViolation` / `NoSuchTool` / `JsonParseFailed`).
- **Error wrap — `app/query/src/tool_call_preparer.rs::prepare_one_pending_tool_call`.** `tc.invalid` → synthetic `tool_result(is_error: true, "<tool_use_error>{prefix}: ...</tool_use_error>")`. The next turn carries the structured error back to the model and it self-corrects — no LLM repair callback, no static retry; recovery is the agent loop itself.

If you find yourself adding tool-input parsing or validation to `vercel-ai/ai/src/generate_text/`, you almost certainly want `app/query` instead. **Why validation lives in `app/query`, not here:** `coco-inference` is deliberately tool-agnostic (no `coco-tool-runtime` dep, no schema registry); every other runtime caller passes `tools: None`. Validation sits at the only path that executes tools, where `ToolSchemaValidator` is already on `ToolUseContext`. Wire contracts are locked by `vercel-ai-*/tests/*_wiremock.rs`; end-to-end validation coverage lives in `app/query/tests/tool_input_error_chain.rs` + `app/query/src/tool_input_validate.test.rs`.

**Anthropic streaming parse-once seam:** the adapter parses accumulated `input_json_delta` only at `content_block_stop` (also injecting provider tool `type` for code-execution variants). Repairable input becomes `ParsedJson` at the inference seam; unrecoverable/non-object input becomes `UnrecoverableRaw` + `JsonParseFailed`, and `app/query` skips schema validation so the model sees the parse-failure prefix, not a secondary schema error. Freeform/raw-string tools are decided by `coco-tool-runtime::ValidatedInput` coercion, not provider guessing.

## Design Notes

- **`extra_body` overrides typed writes by design — final-merge priority (F1 doctrine).** Single source of truth; every per-provider options doc points here.
  - Per-call pipeline: (1) typed channel writes the typed body (e.g. `body.generationConfig.thinkingConfig`), then (2) `provider_options[<ns>]` extras **deep-merge** over it via `vercel_ai_provider_utils::merge_json_value`. Extras ALWAYS win — the user/integrator escape hatch.
  - `thinking_convert.rs` is a typed-channel producer using extras as transport; when its keys overlap a typed write at the same JSON path, extras win by deep-merge contract — intentional, not a coordination bug; no plan to collapse the two writers.
  - Producers augmenting a sibling typed write MUST emit the **nested wire-correct shape** (e.g. `{generationConfig: {thinkingConfig: {includeThoughts: true}}}`) — a flat root-level key would clobber sibling typed writes after deep-merge.
  - **Null in extras is a no-op** — `merge_json_value` treats `Value::Null` as "preserve base" at every depth. To unset a key, omit it from extras (do NOT pass `null`).
- **`ProviderOptions` extraction is uniform and fail-fast.** Every per-provider options struct implements `vercel_ai_provider_utils::ExtractExtras`, parsed via `extract_namespaced(po, canonical_ns, custom_ns)`.
  - Deep-merges canonical + custom namespaces per-key (custom wins), deserializes the typed struct, splits `#[serde(flatten)] extra` automatically. Missing namespaces → typed defaults; unknown keys → extras; malformed typed fields → `ExtractNamespacedError` aborts request-body construction instead of silently dropping config.
  - `openai-compatible`'s 3-level camelCase fallback (`camel(provider)` → `raw(provider)` → `openaiCompatible`, in `provider_options_key::get_effective_provider_options`) is the documented exception; the typed/extras split still uses `ExtractExtras`.
  - `shallow_merge_object` no longer exists — every adapter deep-merges via `merge_json_value` so nested user paths compose with typed writes.
- **Thinking-level conversion (`thinking_convert`):** `ThinkingLevel` → per-provider `ProviderOptions`; signature `to_extra_body(level, api, capabilities: &[Capability])` (`build_call_options` threads `info.capabilities`). `level.options` passes through unconditionally.
  - Anthropic arm (exhaustive over `ReasoningEffort`): `Disable` → `thinking: {type: disabled}`; `Auto` → `{type: adaptive}` **only when `capabilities` contains `Capability::AdaptiveThinking`** (registry-driven gate; models without it degrade to server default, preventing 400s); `Minimal` maps to `Low`; explicit levels emit BOTH legacy `thinking: {type: enabled, budgetTokens?}` AND `output_config.effort` (`XHigh` → `"max"`). `budget_tokens` is faithfully forwarded — `None` omits the key end-to-end; endpoints requiring one must declare it at the `ModelInfo` layer. The `output_config` write rides the extras path and never sets `AnthropicProviderOptions.effort`, so the `effort-2025-11-24` beta is not added — opt in via `provider_options["anthropic"]["effort"]`. Adapter-side details (budget non-fallback, adaptive pre-gating, beta headers): `vercel-ai/anthropic/CLAUDE.md` "Extended thinking".
  - OpenAI arm: emits `reasoningSummary: "auto"` for `Auto` **and** every explicit level (only `Off` is skipped) — requesting the reasoning *summary* is independent of effort; coupling it to `is_explicit_level()` silently dropped reasoning text for models whose builtin default level is `Auto`.
  - Gemini / OpenaiCompat arms keep the `is_explicit_level()` gate (`Disable`/`Auto` emit nothing); the capability slice is unused there.
- `ApiClient` is crate-private. Public callers go through `ModelRuntimeRegistry`; model routing (`ModelRole` → `ModelSpec`) and explicit provider/model selection resolve there.
- **Cache-strategy emission is pass-through, not policy.** This crate emits the typed signals (`cacheStrategy`, `requestedBetas`, `agenticQuery`, `querySource`) into `provider_options["anthropic"]`. All decisions (1h-TTL eligibility latch, allowlist match, marker placement, beta gating) live in `vercel-ai-anthropic` (`cache_policy`, `cache_placement`, `beta_resolver`). The raw map lands in merged `extra_body` verbatim — no re-encoding hop.
- **1M-context credits clamp is runtime state, not text parsing.** `ModelRuntime` latches `InferenceError::is_long_context_credits_required()` and clamps future `ModelRuntimeSnapshot.model_info.context_window` values above `200_000` down to `200_000` for the session. The adapter detects the wording; inference only consumes the typed flag.
- **Per-call fallback window floors are opt-in.** Ordinary calls keep the configured fallback chain; compact summarization sets `fallback_min_context_window` to avoid falling to a smaller-window model mid-summary.
- **Detector hashes the merged map.** `build_call_options_with_extra` snapshots the merged map BEFORE namespace-wrapping; `client::build_prompt_state_input` hashes the snapshot — new pass-through keys auto-track without touching the detector.
- **Single typed `StopReason` for the whole workspace.** `coco_inference::StopReason` re-exports the extended `vercel_ai_provider::UnifiedFinishReason` (8 variants — `EndTurn`, `StopSequence`, `ToolUse`, `MaxTokens`, `ContextWindowExceeded`, `ContentFilter`, `Error`, `Other`). Mapped exactly once at the provider-adapter seam; higher layers match on the enum with zero wire-string parsing. See `vercel-ai/provider/src/language_model/v4/finish_reason.rs` for the multi-LLM mapping table.
- **`QueryResult.stop_reason` and `StreamEvent::Finish.stop_reason` carry the full `FinishReason` struct** (`{ unified, raw }`), propagated verbatim from the provider seam — never decomposed into bare enum + sibling raw string.
  - Behavioral code matches on `.unified` (or the delegating `is_*` helpers); `.raw` is an **in-memory** diagnostic for every variant, read by the side-query `Other` mapping (`app/cli`) and surfaced in logs via `Display` (`other(compaction)`).
  - `raw` is never persisted — `FinishReason` is `#[serde(transparent)]` over `unified`; persisted types (`AssistantMessage`, `CompletedOutcome`, `ResponseLog`) store the bare `StopReason` projection, which `app/query` projects at `engine_stream_consume` after logging the raw.
- **Abnormal `stop_reason` escalates to `warn`, never to error.** Both the blocking client (`client.rs::query`) and the streaming pipe (`stream.rs::stream_event_from_part`) emit a `warn!` when the finish reason is abnormal (see the `is_abnormal()` methods on `UnifiedFinishReason` / `FinishReason` — complement of `EndTurn` / `StopSequence` / `ToolUse`), carrying `query_source`, `tokens_out`, and (blocking) `max_tokens`.
  - **Not an error:** `stop_reason` is a result field. `app/query` relies on receiving the typed variant to dispatch recovery — `MaxTokens` drives 2-phase `MaxOutputTokensEscalate` / `MaxOutputTokensRecovery`; `ContextWindowExceeded` routes to `handle_context_overflow` (reactive compaction). Returning an `InferenceError` would break recovery and discard partial content.
  - Callers that only need text (e.g. `tool_use_summary` side-fork) re-warn at their layer when the stop is abnormal AND extracted text is empty.
