## Review pipeline

1. Build a concise overview from the PR title, body, branch names, labels, and changed file list.
2. Inspect the PR diff for correctness, regressions, missing tests, security issues, performance issues, and project-convention mismatches.
3. Verify every potential finding against the diff before reporting it. Do not report style-only or speculative concerns.
4. Prioritize actionable findings by severity, with file and line references when possible.

Present the review with:

- A short PR overview.
- Findings first, ordered by severity. For each finding, include `file:line - summary`, the concrete failure scenario, and the suggested fix. If there are no findings, say so clearly.
- Any test gaps or residual risks that remain after the review.
